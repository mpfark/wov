
# Offscreen Boss Kill: Reward & Death-Cry Fix

## What happened in your session

Two separate kill paths exist, and they were out of sync:

1. **`combat-tick`** runs while a player is at the creature's node (your party-mate at the boss). It only knows about party members **present at that node**. You (Aelthir) were dead and at a different node, so you were not counted as a recipient. It also broadcasts its kill log to all party members via the `party-combat-{id}` channel.
2. **`combat-catchup`** runs from your client when you leave a node with active DoTs. When your DoT killed the boss offscreen, this path awarded XP/gold/BHP to **you and your party-mate via `award_party_member` RPC** — but it never:
   - sent a reward log message to your party-mate
   - refreshed your party-mate's in-memory character state (their UI showed stale XP until relog)
   - emitted the boss `boss_death_cry` to anyone

So in your case, the kill very likely landed in **both paths** in some order:
- One of the two killed it; the other returned a stale "creature_states" with the boss already dead.
- Your party-mate's `combat-tick` produced the "transcends experience" line (because the only at-node member was level-capped). That line was broadcast over `party-combat-{id}`, which is why your party-mate saw it. Your client never logs `combat-tick` results because you weren't in the party tick — that's why it was missing for you.
- Your offscreen-DoT path silently awarded the missing XP via the RPC. Your party-mate's client never refreshed, so the XP only appeared on relog.
- The boss death cry was only generated by `combat-tick` (which had no kill to attribute, since the DoT killed it). `combat-catchup` doesn't emit death cries at all.

## Fix plan

### 1. `combat-catchup` — broadcast rewards & death cry to party members

In `supabase/functions/combat-catchup/index.ts`, after awarding `kill_rewards` (around lines 350–383):

- For each unique recipient party, send a `party_reward` broadcast on `party-broadcast-{party_id}` for **each non-source recipient** (so their UI refreshes character data — see `usePartyBroadcast.ts` listener).
- For each killed creature where `creature.rarity === 'boss'` and `creature.boss_death_cry` is present, send a `world` broadcast on `world-global` with `kind: 'boss_death'`, the substituted text, and `actor` set to the primary source character's name. This mirrors the `useGlobalBroadcastSender` payload shape so all online players see the world emote.
- Also send a `party_combat_msg` broadcast on `party-broadcast-{party_id}` containing the kill summary (so party-mates at other nodes see "☠️ Boss has been slain by DoT! +XP / +Gold each."). Include a stable `id` (e.g., `${creatureId}:catchup`) and the source character's name as `character_name`. The existing `processIncomingLog` in `GamePage.tsx` will display it.

The edge function uses the service-role client. Use `db.channel('world-global').send({...})` and `db.channel('party-broadcast-' + party_id).send({...})` — Supabase Realtime accepts service-role broadcasts without `subscribe()` when sending only.

### 2. `combat-catchup` — emit a `kill_rewards` entry to the source player too

The current path already returns `kill_rewards` to the calling client, which `useOffscreenDotWakeup.ts` logs locally. That part is fine — but expose `boss_death_cry_text` and `source_character_name` on each entry so:
- The source's local log can show "🌫️ <world emote>" via `eventBus.emit('log', { message: ... })` next to the existing "slain by DoT" line.
- The source client can additionally call `useGlobalBroadcastSender` to fan the death cry out to all online players (defense in depth in case the edge-function broadcast is dropped).

### 3. Avoid double-awarding when both paths fire

Today `combat-tick` and `combat-catchup` could both decide to "kill" the same boss in a narrow race window:
- `combat-tick` removes the creature's effects from `active_effects` once the creature dies (it deletes effects keyed by `killedCreatureIds` in `cleanupEffects`).
- `combat-catchup` reads `is_alive = true` creatures only (line 178) and resolves only effects for those. So if `combat-tick` wins the race, `combat-catchup` finds no alive boss and does nothing — good.
- If `combat-catchup` wins, it sets `is_alive = false` via `writeCreatureState`; the next `combat-tick` sees the creature gone and never awards.

So no real double-award risk — but to make it bullet-proof, in `combat-catchup` skip awarding for any creature in `cKilled` whose `is_alive` was already `false` when re-read at the very end (one extra `select id, is_alive from creatures where id in (...)` before `award_party_member`). This guards against the edge case where `combat-tick` raced ahead between our SELECT and writes.

### 4. Client: `usePartyBroadcast.ts` already handles `party_reward` — keep as-is

`GamePage.tsx` already listens for `partyRewardEvents` and refetches the character on each new event (lines 451–463). With step 1 in place, your party-mate's XP/gold will refresh in real time — no relog needed.

### 5. No changes needed to `useOffscreenDotWakeup.ts` reward log

It already logs the "☠️ {creature} has been slain by DoT! …" line locally. Step 2 just adds the world-emote line right after.

## Files to change

- `supabase/functions/combat-catchup/index.ts`
  - Add `party_reward` broadcast per recipient party member (skip source).
  - Add `party_combat_msg` broadcast per party with the kill summary.
  - Add `world-global` broadcast for boss death cries.
  - Append `boss_death_cry_text` and `source_character_name` to each `kill_rewards` entry.
  - Optional safety: re-check `creatures.is_alive` before awarding.

- `src/features/combat/hooks/useOffscreenDotWakeup.ts`
  - When a returned `kill_rewards` entry has `boss_death_cry_text`, emit an additional `log` event with `🌫️ {text}`.
  - Optionally also call the global broadcast sender as a redundancy net (acceptable to skip if step 1 broadcast is reliable).

No DB schema or RLS changes are required.

## Result

After the fix, when a boss dies offscreen via your DoT while your party-mate is elsewhere:
- Your party-mate sees the slain message in their combat log immediately.
- Their XP/gold updates in the UI without a relog.
- The boss world emote is broadcast to every online player.
- Your local log shows both the kill line and the world emote.
